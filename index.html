1 DNS解決（住所を調べる）
google.com
🔍 介護美容 資格
📒
住所を調べています...

よし子のiPhone（Safari）が 📒DNSリゾルバ に尋ねます。

「google.com の住所を教えて？」（UDP通信・ポート53）

問い合わせ先はキャリアのDNS、または 8.8.8.8（Google Public DNS）。

キャッシュになければ、順番に聞いていきます:

ルートDNS → .com DNS → google.com DNS

そして答えが返ってきます:

「142.250.196.110 だよ」（IPアドレスを返却）

DNSの階層構造について
DNSはツリー構造になっており、ルートDNS（.）→ トップレベルドメインDNS（.com）→ 権威DNSサーバー（google.com）と順番に問い合わせが行われます。一度解決したアドレスはキャッシュされるため、2回目以降はほぼ一瞬で解決されます。
2 TCP接続（道をつなぐ）

住所がわかったので、次はサーバーと「道」をつなぎます。この接続には3回のやりとりが必要です。

1 SYN → 「つながっていいですか？」
2 ← SYN-ACK 「いいですよ、こちらもいいですか？」
3 ACK → 「いいですよ！」
3ウェイハンドシェイクとは
この3回のやりとりを「3ウェイハンドシェイク」と呼びます。絵本では「3回おじぎ」のイメージです。お互いに「準備OK」と確認し合ってから、はじめてデータのやりとりが始まります。
3 TLS暗号化（秘密の約束）

「https:// だから暗号化しましょう」

1 ClientHello → 「暗号方式のメニューです」
2 ← ServerHello + 証明書 「この方式でいきましょう。身分証です」
3 鍵交換 → 共通の秘密鍵を生成

以降の通信はすべて暗号化されたトンネルの中を通ります。

TLSを絵本で言うと
TLSハンドシェイクは、「二人だけの秘密の言葉を決める儀式」です。この儀式が終わると、以後のやりとりは第三者に読まれない秘密の手紙になります。URLバーの鍵マーク🔒は、この暗号化トンネルが有効であることを示しています。

fbclidの正体と役割
fbclid パラメータ 
fbclid=IwAR3xxxxxxxxxxxxxx

これは Meta が全ての広告クリックに自動付与する識別子です。

fbclidの3つの役割:

1. ブラウザ側: Meta Pixel が fbclid を読み取り、_fbc Cookieに保存
2. サーバー側: sGTM経由で Meta CAPI が fbclid を fbc パラメータとして送信
3. Meta側: 「広告クリック」と「その後の行動（CV等）」を紐付け

当社の計測基盤では:

Web GTM → GA4タグ → sGTM（Stape） → Meta CAPIタグ → fbc パラメータに fbclid を含めてMetaサーバーへ直接送信

これにより iOS 14.5+ の Cookie制限を迂回し、正確なコンバージョン計測を実現しています。

UTMパラメータとは
utm_source=ig → 流入元: Instagram
utm_medium=paid → 種別: 有料広告
utm_campaign=kaigo_beauty_2026 → キャンペーン名

→ GA4 で「どの広告から来たか」を分析するための情報です。

🏭
「当社の計測基盤の価値」

なぜsGTM + Stape + CAPIが必要なのか。ブラウザの制約が厳しくなる中、サーバーサイドで正確にデータを捕捉する当社の計測基盤が、クライアントの広告投資を守っています。

💎
「1CVの重み」

1件のコンバージョンの裏側で動く何十ものサーバーと通信。よし子さんの「資料請求」ボタン1つで、DNS、CDN、Meta、Framer、sGTM、GA4、Google Ads――すべてが連携して動いています。

🛡️
「プライバシーと計測の両立」

iOS制限（ITP/ATT）下でも正確な計測を実現する当社の技術力。Cookie Keeper、Stape Store、Dup Checker――プライバシーを守りながらデータを活かす仕組みが、私たちの競争力の源泉です。
